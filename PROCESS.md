1. תכנון המערכת
לפני תחילת הפיתוח, הגדרתי ארכיטקטורת Full Stack המפרידה בין עיבוד הנתונים הכבד לבין ממשק המשתמש:

Backend (Python & Flask): בחרתי ב-Flask בשל היותה ספריה קלה וגמישה המאפשרת אינטגרציה מהירה עם ספריות AI וטיפול בקבצי אודיו. השרת מנהל את קבלת הקבצים, הפעלת מודל התמלול, ותקשורת מול ה-LLM.

Frontend (React): הממשק תוכנן להיות נקי ואינטואיטיבי, עם דגש על חווית משתמש בעברית (RTL). הוא מנהל את מצבי הטעינה (Loading) ומציג את התוצאות בצורה היררכית: סיכום מנהלי בפורמט Markdown ולאחריו התמלול המלא.

זרימת עבודה: העלאת קובץ -> שמירה זמנית בשרת -> תמלול (Whisper) -> ניתוח טקסטואלי (Gemini) -> החזרת JSON למשתמש -> אפשרות לייצוא ל-Word.


2. שימוש ב-AI בתהליך הפיתוח + דוגמא לפרומפט
השתמשתי בבינה מלאכותית כ"שותף לקידוד" כדי להאיץ את הפיתוח ולהבטיח איכות:

בניית הלוגיקה ב-React: נעזרתי ב-AI כדי לבנות את הטיפול ב-FormData וב-Axios לצורך העלאת קבצים מה-Frontend ל-Backend.

כתיבת הפרומפטים: השתמשתי ב-AI כדי לזקק את ה-System Prompt כך שיפיק פלט בפורמט Markdown מובנה (טבלאות וכותרות) שמתממשק בצורה חלקה עם ה-UI.

דוגמה לפרומפט ששימש אותי: "עזור לי לכתוב פונקציה ב-Flask שמקבלת קובץ אודיו, שומרת אותו זמנית, ומחזירה תמלול תוך שימוש בספריית openai-whisper."

3. אתגרים ופתרונות
במהלך הפיתוח נתקלתי במספר אתגרים טכניים שדרשו חשיבה עצמאית:

בעיית ה-CORS: בתחילה, הדפדפן חסם את התקשורת בין React (פורט 3000) ל-Flask (פורט 5000). פתרתי זאת על ידי הטמעת flask-cors בשרת.

זיהוי מודל Gemini: נתקלתי בשגיאת 404 בעת ניסיון להשתמש במודל gemini-pro. פתרתי זאת על ידי תשאול ה-API של Google AI Studio לזיהוי המודלים הזמינים ועדכון הקוד לשימוש ב-gemini-flash-latest, שהתגלה כמודל המהיר והיציב ביותר עבור המשימה.

מעבר לתמלול מקומי: בשל עלויות ה-Whisper API, בחרתי בפתרון חכם יותר של הרצת ספריית openai-whisper באופן מקומי על השרת בשילוב FFmpeg. זה הצריך הגדרות סביבה מורכבות יותר אך חסך עלויות API ואיפשר פרטיות נתונים גבוהה יותר.

4. ה-System Prompt
הפרומפט תוכנן בקפידה כדי להוציא את המקסימום ממודל השפה.

הפרומפט המלא:

"אתה עוזר ניהולי חכם. נתח את תמלול הפגישה הבא בעברית והוצא דוח מובנה הכולל: 
1. כותרת לפגישה. 
2. סיכום תמציתי של עיקרי הדברים. 
3. רשימת משתתפים (אם הוזכרו). 
4. החלטות שהתקבלו. 
5. רשימת משימות לביצוע (Action Items). 
השתמש בפורמט Markdown ברור."

הסבר על מבנה הפרומפט:

הנחיית תפקידים: הגדרת המודל כ"עוזר ניהולי" מבטיחה שפה רשמית ומקצועית המתאימה לסיכומי פגישות.

אילוצי מבנה: פירוט 5 הסעיפים מוודא שהמודל לא "ישכח" את רשימת המטלות או המשתתפים, גם בתמלולים ארוכים.

פורמט Markdown: בחירה זו מאפשרת ל-Frontend להציג את המידע בצורה ויזואלית (בולדים, טבלאות) ומקלה על הייצוא למסמך Word תוך שמירה על היררכיה.

5. זמן ביצוע בפועל
סה"כ זמן עבודה: כ-5.5 שעות.

הזמן חולק בין הגדרת השרת ותמלול האודיו (3 שעות), פיתוח ה-UI ב-React (2 שעות) ותיעוד התהליך (חצי שעה).
